# ADR 001: Use positioned I/O (FileExt::read_exact_at / write_all_at) for all page-level access

## Status

Accepted

## Context

I am building a page-based storage engine. The initial naive implementation would:

- `seek()` to the desired page offset
- then `read_exact()` / `write_all()`

This approach has several serious drawbacks in a database context:

1. **Two syscalls per page access** (`lseek` + `read` / `write`) instead of one
2. **Share file cursor race condition** - multiple threads calling `seek` + read/write can interfere with each other
3. **No atomicity** with respect to concurrent processes 
4. **Error-prune code** - easy to forget `seek`, leave cursor in wrong position, etc.

The rust standard lib provides `std::os::unix::fs::FileExt`, which exposes:

- `read_at` / `read_exact_at`
-  `write_at` / `write_all_at`
- vectored variants

These map directly to the POSIX `pread` / `pwrite` system calls (on Linux, macOS, *BDS, etc.).

## Decision

**I will be using positioned I/O methods (`read_exact_at`, `write_all_at`) for all page reads and writes in `StorageManager`.**

## Consequences

### Positive

- Single syscall per page access -> better random I/O performance
- Thread-safety: no race on the file descriptor's seek position -> `&self` methods become safe to call concurrently (only content-level locking needed later)
- Cleaner code; offset calculation is explicit and local for each call
- Matches industry practice (SQLite, PostgreSQL, LMDB, WiredTiger, reddb, sled, all used positioned I/O for data files)
- Enable future patterns: concurrent writes, mmap alternatives, direct I/O experiments

### Negative

- Unix-only today
    - I accept this limitation for v0
    - Cross platform story can be addressed later

- Slightly more verbose than than plain `read` / `write`
    - Offset calculation must be done manually every time
    - Mitigated by helper method `page_offset(page_id)`
